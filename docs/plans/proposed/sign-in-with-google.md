# Sign in with Google — Implementation Plan

## Context

The app currently requires users to paste a Listen Key from the web. This causes friction and context-switching. Sign in with Google replaces this with a one-tap native sign-in that acquires the Listen Key automatically via a new server endpoint (`POST /api/v1/listener/auth/google`). The paste flow stays as a fallback.

The server endpoint (`POST /api/v1/listener/auth/google`) is live. The OpenAPI spec is published at github.com/jbaddnz/mixmates-listener-api.

---

## Architecture

```
TokenEntryScreen
    │ creates GoogleSignInHelper (needs Activity context)
    │ passes to ViewModel
    ▼
TokenEntryViewModel
    │ 1. Generate nonce
    │ 2. GoogleSignInHelper.signIn(nonce) → Google ID token + name
    │ 3. AuthRepository.signInWithGoogle(idToken, nonce, name) → bearer token + listenEnabled
    │ 4. If listenEnabled: AuthRepository.saveToken() → stored in EncryptedSharedPreferences
    ▼
Same downstream flow as paste (AuthInterceptor, ListenerApi, etc.)
```

Nothing downstream changes. The bearer token is the same regardless of how it was obtained.

## New Files

### 1. `data/auth/GoogleSignInHelper.kt`

Plain class (not Hilt — needs Activity reference). Wraps Credential Manager API.

- `signIn(nonce: String): GoogleSignInResult` — presents One Tap sheet, returns idToken + displayName
- Uses `GetGoogleIdOption` with `setFilterByAuthorizedAccounts(false)` to show all accounts
- Sets `setNonce(nonce)` for replay protection (server validates the nonce)
- Server client ID from `BuildConfig.GOOGLE_WEB_CLIENT_ID`
- Throws on failure — ViewModel handles exceptions
- Nonce generated by the ViewModel (UUID) and passed to both the helper and the API request

Not Hilt-injected because Activity references shouldn't live in singletons. Created with `remember` in the composable, passed to the ViewModel per-invocation.

### 2. `res/drawable/ic_google.xml`

Google "G" vector drawable for the sign-in button. Required for Google branding compliance.

### 5. `data/api/dto/ApiModels.kt`

Add two DTOs at the bottom:
```kotlin
@Serializable
data class GoogleSignInRequest(
    @SerialName("id_token") val idToken: String,
    val nonce: String,
    val name: String? = null,
)

@Serializable
data class GoogleSignInData(
    val token: String,
    @SerialName("is_new_account") val isNewAccount: Boolean,
    @SerialName("listen_enabled") val listenEnabled: Boolean,

```

`name` is optional per the API spec — the server extracts email from the JWT `sub` claim directly.

### 6. `data/api/ListenerApi.kt`

Unauthenticated — AuthInterceptor already skips the header when no token is stored.

### 7. `data/repository/AuthRepository.kt`

Add one method:

Thin wrapper that builds the request DTO and calls `api.signInWithGoogle()`. Consistent with existing style.

### 8. `ui/auth/TokenEntryViewModel.kt`

- Add `isGoogleSigningIn: Boolean` to `TokenEntryUiState`
- Add `signInWithGoogle(helper: GoogleSignInHelper)` method:
  1. Generate nonce (UUID)
  2. Call `helper.signIn(nonce)` → Google ID token + name
  3. Call `authRepository.signInWithGoogle(idToken, nonce, name)` → token + listenEnabled
  4. Check `listenEnabled` from response (no `/auth/me` round trip needed)
  5. If not enabled: don't save token, show upgrade message
  6. If enabled: save token via `authRepository.saveToken()`, set `isValid = true` → triggers navigation
- Catch `GetCredentialCancellationException` silently (user dismissed)
- Catch other exceptions → show error, suggest paste flow fallback

## Edge Cases

| Scenario | Handling |
|---|---|
| No Google Play Services | Credential Manager throws, catch block shows error + suggests paste flow |
| User has multiple Google accounts | Credential Manager handles natively — shows account picker |
| Google identity creates free account | `listenEnabled` in response is false → show upgrade message, don't save token |
| User dismisses sign-in sheet | `GetCredentialCancellationException` caught silently, reset state |
| Token from paste, then signs in with Google | New token overwrites old one — correct behaviour |
| Device rotation mid-sign-in | `viewModelScope` cancels the coroutine, user retries |

## Confirmed with MMC

All questions answered — see updated brief at `docs/mmc-brief-siwg-listen-key.md`:

1. **Google Client ID** — use the existing web client ID (`GOOGLE_CLIENT_ID`). No separate Android credential needed.
2. **Unauthenticated** — endpoint excluded from auth middleware.
3. **`listen_enabled` in response** — included, avoids `/auth/me` round trip.
4. **Error shape** — standard `{ "error": { "code": "...", "message": "..." } }` envelope.
5. **Rate limiting** — 5/min per IP.
6. **Nonce** — yes, use `setNonce()`. Server validates for replay protection (KV with 5-min TTL).

## Feedback for iOS Claude

1. **Response shape**: `POST /auth/apple` should return the same shape as `/auth/google`: `{ "data": { "token": "...", "is_new_account": true, "listen_enabled": false } }`. The `listen_enabled` field was added to avoid an extra `/auth/me` round trip — both platforms should use it.

2. **listenEnabled check**: Check `listen_enabled` directly from the auth response. Don't save the token if listen isn't enabled — show upgrade guidance instead.

3. **Settings wording**: Android is changing "Remove Listen Key" to "Sign out". iOS should do the same if it hasn't already — the key-specific language won't make sense once users sign in with Apple/Google.

4. **Nonce**: MMC wants replay protection. iOS should set a nonce on the SIWA request too and include it in the `/auth/apple` request body.
